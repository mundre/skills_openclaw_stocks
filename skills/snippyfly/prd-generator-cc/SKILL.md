# PRD 生成器

本技能帮助生成清晰、结构完整、逻辑严谨的产品需求文档（PRD），让业务相关方都能切实执行。

一份好的 PRD 需要回答三个问题：
1. **为什么** — 我们在为谁解决什么问题？
2. **做什么** — 产品/功能将实现什么？
3. **做多好** — 成功的标准是什么？

写 PRD 最难的地方不是知道模板——而是进行深入思考。你的职责是帮助用户完成这个思考过程，而不仅仅是填写模板。

## 阶段一：信息收集

在开始撰写之前，通过访谈了解以下内容：

- **问题**：存在什么痛点？谁在经历这个问题？没有这个功能时，现状是什么？
- **用户**：主要用户是谁？是否有次要利益相关方？
- **范围**：这是新产品、现有产品中的新功能，还是对现有功能的改进？
- **约束**：时间线？技术限制？平台（Web/移动端/API）？合规要求？
- **成功指标**：如何判断这个功能成功了？是否有目标数据？

提问要有针对性——不要一次性抛出所有问题。如果用户提供了粗略想法，从中提取已有信息，只询问真正缺失的内容。如果已有足够上下文，直接跳到起草阶段。

**信息不足时的处理原则**：合理推测并标注"【推测】"，在文档末尾列出"待确认事项清单"，而不是让流程卡住。

## 阶段二：撰写 PRD

> 💡 **在此阶段开始前**，读取 `references/prd_template.md` 获取带有示例内容的完整结构参考。

使用以下结构。根据功能范围灵活调整各部分深度——一个小型内部工具不需要与旗舰产品功能同等深度。始终使用清晰、无歧义的语言。

### 标准 PRD 结构

```
# [功能/产品名称] — 产品需求文档

**状态**：草稿 / 审核中 / 已批准
**作者**：[姓名]
**最后更新**：[日期]

---

## 1. 前言

1.1 需求基本信息 （必填）
产品经理：[若用户未提供，则填写"【待补充】"]
1.2 文档变更日志 （必填）
版本  日期  变更内容    变更章节    作者
v1.0  [当前日期]  初始创建    全部  [产品经理名称]

---

## 2. 需求概述 （必填）

### 2.1 背景
当前情况是什么？现状如何？为何是个问题？
根据需求类型自动填写：政策类需包括政策发文链接及解读；业务类交代业务背景。若信息不足，请合理推测并标注"【推测】"

### 2.2 用户痛点
谁受到影响，以及如何影响？要具体——泛泛而谈的"用户想要 X"太薄弱。尽可能以数据、研究或具体观察为依据。

### 2.3 业务动机
解决这个问题对业务有何意义？（营收、留存、合规、竞争差异化等）

---

## 3. 目标与非目标

### 3.1 目标
- ✅ 将新用户首次操作时间缩短 30%
- ✅ 允许用户以 CSV 和 PDF 格式导出报告

### 3.2 非目标
- ❌ 不包含批量导入功能（计划于 Q3 推出）
- ❌ v1 版本不支持移动端

---

## 4. 用户故事与使用场景

### 4.1 主要用户故事
格式："作为[用户类型]，我希望[执行操作]，以便[达成结果]。"

### 4.2 关键场景 / 流程
**正常路径：**
1. 用户导航至...
2. 用户点击...
3. 系统响应...

**错误/边界情况：**
- 如果用户尚无数据，显示空状态并提供引导操作...

> 💡 若本功能涉及 3 个以上用户阶段或多个角色，建议配套绘制用户旅程图。
> 参见 `assets/user_journey_example.md` 了解旅程图结构和推荐工具。

---

## 5. 功能需求

要求：
- 编号、无歧义、可测试的系统行为说明
- 每条需求应可独立验证——如果你无法为其编写测试用例，说明它可能过于模糊
- 数据模型需列出字段名、类型、是否必填、描述
- 业务规则需明确边界条件
- 若用户输入信息不足，标注"【待补充】"并在末尾列"待确认事项清单"

5.1 整体需求 （必填）
// 概括本次需求包含的所有功能模块

5.2 功能 （必填）
5.2.1 优先级排序
// 列出功能点优先级（P0/P1/P2）

对每个功能点，按以下子结构展开：
5.2.x 功能点名称
  5.2.x.1 功能描述
  5.2.x.2 使用场景
  5.2.x.3 状态流转逻辑
  5.2.x.4 权限控制
  5.2.x.5 异常处理规则  ← 参见 ux_guidelines.md 第 6 节
  5.2.x.6 数据模型结构（字段级说明）
  5.2.x.7 业务规则（可测试，格式：FR-XXX）

5.3 接口需求
若涉及系统对接，每个接口需包含：
  接口名 / 描述 / 请求参数表 / 响应参数表 / 错误码说明

## 6. 非功能需求

约束系统如何运行的需求，而非系统做什么。

- **性能**：页面在 4G 网络下须在 2 秒内加载完成
- **可用性**：99.9% 正常运行时间 SLA
- **安全性**：传输中的所有数据必须加密（TLS 1.2+）
- **无障碍**：符合 WCAG 2.1 AA 标准（具体标准见 `references/ux_guidelines.md` 第 9 节）

仅包含与本功能真正相关的内容。
```

## 写作原则

**具体，而非空泛。** "用户能导出数据"太弱。"用户可将筛选后的数据导出为最多 10 万行的 CSV 文件"才是可测试的。

**为读者而写。** 工程师需要足够的细节来构建。设计师需要背景来做好决策。高管需要快速理解"为什么"。相应调整内容深度。

**区分做什么与怎么做。** PRD 定义需求，而非实现方案。"系统应在 5 秒内完成上传处理"——可以。"系统应使用 Redis 进行任务队列"——不可以（除非有充分理由）。

**明确非目标。** 未言明的假设会导致范围蔓延。明确说明你不做什么。

**用证据支撑观点。** "用户对 X 感到沮丧"如果有数据支撑则更有力："在 Q2 调研中，68% 的受访者将 X 列为最主要的痛点。"

**引导而非填充。** 当用户信息不足时，标注"【待补充】"并列出待确认事项，帮助用户意识到遗漏，而非自行填造内容。

## 输出格式

如有可能，将完整 PRD 生成为 `.docx` 文件（检查 docx 技能是否可用），否则输出为格式良好的 Markdown 文档。文档应可直接分发给利益相关方——清晰的标题、合理的逻辑流程、没有未完成的部分。

### 样式要求

- **正文**：字体为宋体（SimSun），字号为小四（12pt）
- **段落行间距**：1.5 倍（line-height: 1.5）
- **段落首行缩进**：每个段落首行缩进 4 个字符（text-indent: 4em）
- **表格文字**：字体为宋体（SimSun），字号为五号（10.5pt）
- **表格对齐**：所有表格居中显示

当用户对某些细节不确定时，提供合理的默认值，或将该部分标注为"待定——[需解决的问题]"，而不是凭空填写内容。

---

## 语气与风格

- 主动语态，需求使用现在时（"系统允许..."而非"系统将允许..."）
- 需求编号便于追溯（FR-001、FR-002 等）
- 段落简短；避免大段文字
- 描述用户流程时，编号步骤优于叙述性文字
- 比较、指标、风险使用表格展示
