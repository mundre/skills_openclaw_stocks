# GitHub 项目开发标准流程

---

## Abstract

> 小白程序员的完整开发/执行/验收方法论。核心理念：先定义问题，再定义改法，再写代码，再做验证，最后才发布。

---

## Overview

### 🎯 核心原则

**一句话记住**：
> **先定义问题，再定义改法，再写代码，再做验证，最后才发布。**

### 🚫 常见错误

- ❌ 直接跳到第 3 步开始写代码
- ❌ 把"修 bug"当成"顺便重构"
- ❌ 不看旧版本，凭记忆重写
- ❌ 一边修 bug，一边改风格
- ❌ 只做"能运行"测试，不做行为验证
- ❌ release note 与实际代码不一致

---

## Details

### Phase 0：接单前先分类

| 任务类型 | 目标 |
|---------|------|
| **Bug 修复** | 最小修复 |
| **小功能新增** | 最小可用实现 |
| **重构** | 行为不变、结构变好 |
| **兼容性修复** | 输入输出格式、接口、版本适配 |
| **发布收尾** | 事实与文档一致 |

---

### Phase 1：需求澄清模板

**每个任务必须写清"做什么"和"不做什么"**

```text
【任务类型】Bug 修复
【问题现象】例如：content_preserved 统计的是全部消息，不是实际保留消息
【正确行为】只统计 kept_older + kept_recent
【影响范围】compress() / CompressionStats / report 输出
【非目标】不修改摘要逻辑，不修改 CLI，不重构其他模块
```

---

### Phase 2：改动设计模板

**在动手前，先说明"要改哪里，不改哪里"**

| 项目 | 内容 |
|------|------|
| 目标文件 | 例如 `scripts/lobster_press_v152.py` |
| 目标函数 | 例如 `compress()` |
| 需要修改 | 删除错误统计块；新增正确统计块；复用 `kept_recent` |
| 不应修改 | `_generate_summary()`、CLI、parser 结构 |
| 风险点 | tuple/list 结构变化可能影响 `output_lines` 和统计循环 |
| 验证方式 | `py_compile` + 构造样例 + diff 检查 |

---

### Phase 3：建立基线版本

#### ✅ 正确做法
```bash
git checkout v1.5.4
cp scripts/lobster_press_v152.py scripts/lobster_press_v155.py
```

#### ❌ 错误做法
- 不看旧版本，直接新写一个文件
- 记忆里觉得"差不多"，就自己拼
- 上一版还没验证，就继续叠补丁
- 一边修 bug，一边把整个文件风格都改了

---

### Phase 4：编码原则

#### 黄金法则

**一次提交只解决一个问题集合**

#### 8 条编码纪律

1. **先复制旧代码，再局部替换，不要凭记忆重写**
2. **改函数前，先通读函数的输入、输出、副作用**
3. **涉及数据结构变化时，先搜所有使用点**
4. **不要同时改逻辑和风格**
5. **不要在 bug fix 里做重构**
6. **不要修改未被需求要求的行为**
7. **不要在没有验证前说"修好了"**
8. **不要让 release note 超前于实际代码**

## 测试方法：4 层验证

### 1. 语法验证
```bash
python3 -m py_compile scripts/xxx.py
```

### 2. 导入验证
```bash
python3 -c "from scripts.xxx import ClassName; print('ok')"
```

### 3. 行为验证（核心）
- 构造最小样例输入
- 验证修复点
- 验证输出符合预期

### 4. 回归验证
- 检查旧功能是否仍正常
- dry-run 还能不能跑
- CLI 行为是否符合预期

---

## Diff 审查：3 件事

### 1. 改动量是否匹配任务规模

| 任务类型 | 预期改动量 |
|---------|-----------|
| 修 1 个小 bug | 5–30 行 |
| 修 1 组相关 bug | 20–80 行 |
| 小功能新增 | 30–150 行 |
| **超过 200 行** | **必须怀疑自己是否改多了** |

### 2. 是否改到了非目标区域

- parser 大改（但任务只涉及 compress）
- CLI 大改（但任务只涉及统计）
- 新文件创建（但任务是 bug fix）

### 3. 发布说明是否和 diff 一致

release note 里写"修了 5 个 bug"，那 diff 里必须能逐条找到对应改动。

## 提交与发布规范

### Commit message 模板

```text
fix: 修复 content_preserved 统计时机错误

- 删除 compress() 开头的全量统计
- 在 kept_older / kept_recent 确定后统计实际保留内容
- 复用 kept_recent，避免重复变量
```

### Release note 模板

```markdown
## 修复内容
- 修复 content_preserved 统计时机错误
- 修复去重使用固定 0.5 分数的问题

## 影响范围
- scripts/lobster_press_v152.py
- compress()

## 验证
- ✅ py_compile 通过
- ✅ 导入测试通过
- ✅ 最小样例验证通过
- ✅ diff 审查完成

## 已知不变更
- 不修改 CLI 参数
- 不修改摘要生成逻辑
- 不新增文件
```

## 验收清单：15 项必答

### A. 需求一致性
- [ ] 我能用一句话说清这次修复的目标
- [ ] 我知道这次"不打算修"的内容有哪些
- [ ] 代码改动与 issue 描述一致

### B. 技术正确性
- [ ] 我基于正确版本开始修改
- [ ] 我没有重写整个文件
- [ ] 数据结构变化已同步所有引用点
- [ ] 新逻辑不会破坏旧逻辑

### C. 测试验证
- [ ] 语法检查通过
- [ ] 导入检查通过
- [ ] 最小样例验证通过
- [ ] 回归测试通过

### D. 发布质量
- [ ] diff 大小与任务规模匹配
- [ ] release note 与实际代码一致
- [ ] 版本号、文档、注释已同步
- [ ] 我可以指出这次改动的风险点

**⚠️ 这 15 项只要有 1 项答不上来，就不要发布。**

## 每日开发流程（9 步）

| 步骤 | 动作 |
|------|------|
| 1 | 读 issue（只理解，不改代码） |
| 2 | 写"5 行任务卡" |
| 3 | 确定基线版本 |
| 4 | 列改动点 |
| 5 | 编码（最小修改） |
| 6 | 本地验证（4 层测试） |
| 7 | 看 diff（确认没偏题） |
| 8 | 写发布说明 |
| 9 | 最后复盘 |

---

## 执行要求：5 项必须说明

以后所有 bugfix / patch 版本，必须明确写出：

1. **任务目标**：这次到底修什么
2. **改动范围**：改了哪些文件 / 函数
3. **非改动范围**：哪些地方明确不碰
4. **验证方法**：做了哪些测试
5. **diff 规模**：总共改了多少行，是否符合预期

**如果做不到这 5 项说明，就不要再写"已修复"。**

---

## 关键能力训练

1. **精准改动能力**：只改该改的地方
2. **验证能力**：能证明修复有效
3. **一致性能 力**：issue、代码、release note 三者一致
4. **收敛能力**：不把 patch 做成重构
