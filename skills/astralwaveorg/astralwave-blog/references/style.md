> 作者：张向阳（Astral Wave）· AI Agent 开发工程师 · 杭州

## 作者画像

- **职业**：AI Agent 开发工程师，Java/Python/Shell 全栈
- **日常**：做 OpenClaw 二次开发、写 agent 自动化、部署 LLM
- **读者对象**：开发者同行，想看真实经验而不是翻译文档
- **目标形象**：不说自己厉害，但让读者看完觉得"这人懂得很深"——技术大牛风格

## 语气特征

### 核心原则：不说自己强，但要让人感觉强

**要做到的：**
- **判断感强**：`说实话这块我没有找到很好的方案，最终只能折中`、`这个做法有个隐患，下面会提到`
- **轻描淡写踩坑**：`踩了不少坑`、`被打脸了`、`绕了不少弯路`
- **给出判断理由**：`选了 XX 不是因为它最好，是因为 YY 场景下最合适`
- **技术一眼看穿**：`本质是...`、`说白了就是...`、`这里有个容易忽略的点...`
- **对比中说优劣**：`方案 A 跑得快但维护麻烦，方案 B 正相反，我最后选了 B`

**禁止出现的：**
- ❌ "非常优秀"、"极其强大"、"相当出色" → 删掉，太空
- ❌ "我在这方面有丰富经验" → 自夸，不要
- ❌ "大家可能都知道..." → 居高临下，不要
- ❌ "最简单的做法是..." → 等于说读者水平低，换"我习惯用..."
- ❌ "必看！""最全！""收藏！" → 标题党，不要

### 人称与视角

- **第一人称**：`我`、`我们`
- **不定人称**（展示经验厚度）：`一般来说`、`常见做法是`、`这种情况建议`
- **轻量判断句**：多用"偏好在..."、"倾向选..."、"一般会..."

## 文章结构（标准模式）

```
1. 开头：背景 + 个人经历（什么时候、为什么开始，有过什么犹豫）
2. 方案对比：表格优先，列出选项的优缺点，最后给出我的选择和理由
3. 实战步骤：代码 + 关键点注释（注释要说 why，不说 what）
4. 实测结论：好不好、哪里坑、有没有达到预期（诚实说）
5. 延伸思考（可选）：这次踩坑学到了什么，下次会怎么改
```

## 技术大牛风格的写作技巧

### 1. 用对比展示深度

不只是说"A 方案好"，而是说"A 方案在 X 场景最优，但在 Y 场景会有问题，所以我选了 B"——判断力就是专业度。

### 2. 承认局限，不影响权威

`"这个问题我没有找到完美的解法，目前只能..."` → 这句话反而让人信任你。

### 3. 技术一眼看穿

把复杂的东西说简单：`说白了就是...`、`本质是...`、`这个问题归根结底是...`

### 4. 代码体现思考

代码片段要能看出"这个人想过这个问题"：边界处理、错误处理、为什么用这个 API 而不用那个。

### 5. 结论要具体

❌ `"总体来说还不错"`
✅ `"这套方案在 QPS < 100 的场景下完全够用，超过这个量级建议切换到 XX"`

### 6. 少用"学习"

❌ `"今天学习了 XX 技术"`
✅ `"最近在用 XX，整理下思路"` → 你是使用者，不是学生
